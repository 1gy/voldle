# Accessibility / i18n

## Principle

**A message reaches several readers at once.** Writing that is kind to visual readers, screen-reader users, and translators converges on the same answer: short, contextually self-contained, free of metaphor.

## Rationale

"Answer is up there" sounds natural to a sighted reader, feels verbose to a screen-reader user, and saddles a translator with a culture-bound metaphor (does "up" mean physical position, or a higher value?). Reduce the message to the comparison itself ("higher") and meaning falls out of context, while the translation key can stay semantic — `hint.higher`.

We keep translation keys semantic rather than literal so each language can use its own natural phrasing. A word-for-word translation between languages often sounds awkward — an English speaker writing "Higher" doesn't construct the same full sentence a Japanese speaker would for the same hint. Pick a granularity that lets each locale read naturally.

A meaningful glyph (↑↓ etc.) hidden from screen readers creates an information disparity — only sighted users get the hint. Use `role="img"` + `aria-label` to give the glyph a name. Prefer `aria-label` over `title` (screen readers reliably announce the former). Conversely, decorative glyphs are silenced with `aria-hidden="true"`.

## Applying

- Meaningful glyphs use `role="img"` + `aria-label`; decorative ones use `aria-hidden`.
- Messages are short, contextually self-contained, free of metaphor.
- Translation keys are semantic (`hint.higher`, `error.notFound`).
- Even when strings are hardcoded today, structure them as a `{ up: "...", down: "..." }` table so introducing i18n is just a matter of swapping in `t()`.
